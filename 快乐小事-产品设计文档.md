## 一、产品理念

### 核心命题
**在碎片化的日常中，帮助人们主动觉察和收集快乐。**

不是大事件记录器，不是日记本，是一个"快乐放大镜"——让你发现那些容易被忽略的、微小但真实的快乐瞬间。

### 设计哲学
1. **轻量先于完整**：记一件事只需要10秒，不要求完整叙事
2. **情绪先于信息**：关注"这件事让我什么感觉"而非"发生了什么"
3. **温暖先于功能**：UI的每一个细节都服务于"让人感到被温柔对待"
4. **本地先于云端**：数据存在你手上，云只是锦上添花的同步工具

### 目标用户画像
- 忙碌的职场人，需要在高压中找到"小确幸"的出口
- 不习惯写长日记，但愿意花几秒记一笔的人
- 对美感有要求，不能接受粗糙工具的人

## 三、功能模块详解

### 模块1：笑一声（哈哈机）

**定位**：进入app的第一个tab，用游戏化的方式给用户一个笑点。

**交互设计**：
- 小女孩SVG角色 + 网兜（橙红色帽子+背带裙）
- 彩色泡泡从天空落下（emoji球+普通泡泡混合）
- 用户滑动屏幕控制小女孩左右移动接泡泡
- 接到emoji球 → 弹出一条随机笑话（180+条冷兔风格）
- 笑话自动存入"泡泡池"（历史收藏）

**设计思考**：
- 为什么放在第一个tab？—— 登录后第一件事是"笑"，建立积极心理暗示
- 为什么是接泡泡不是传统抓娃娃？—— 泡泡更轻盈、更快乐、更低门槛
- 15秒超时保护 —— 防止用户玩太久等不到结果

### 模块2：记好事（快乐广场）

**定位**：核心功能，日历式快乐记录 + 瀑布流浏览。

**交互设计**：
- 顶部固定：标题 + "记录好事"按钮 + 子tab切换
- 快乐广场（默认）：瀑布流双列卡片展示所有记录
- 回看时光：按日期分组的时间线视图 + 日期跳转筛选
- 点击任一条 → 进入全屏详情页（卡片翻页模式）

**记录内容**：
- 文字（必填，200字以内）
- 配图（可选，自动压缩到600px宽）
- 快速标签（预设10种 + 茸言茸语）
- 配文（从100+古诗词中选择）
- emoji装饰（随机分配）
- 天气（自动获取当天深圳天气）

**设计思考**：
- 为什么限制200字？—— 强制简短，降低记录门槛
- 为什么有"快速标签"？—— 有些快乐无法言语，一个标签就够了
- 为什么配文是古诗词？—— 提升格调感，让平凡小事有了诗意

### 模块3：随Ta去（坏情绪粉碎机）

**定位**：情绪宣泄出口。快乐不是压抑负面，而是正确处理后放下。

**交互设计**：
- 纸条式输入框（手写质感）
- 写完点"随Ta去" → 纸条被吸入粉碎机动画（5秒完整过程）
- 碎片飘落 + 粉碎音效（白噪音+碎片敲击3秒）
- 显示随机安慰语（100+条）
- 保存粉碎记录（删除线样式展示）

**设计思考**：
- 为什么动画要5秒？—— 仪式感。太快了没有"告别"的感觉
- 为什么保存粉碎记录？—— 回看时发现"原来当时烦的事现在看来根本不算什么"
- 为什么用"随Ta去"不用"粉碎"？—— 更温和，传达的是"放下"而非"对抗"

### 模块4：问观音（观音灵签）

**定位**：趣味+精神慰藉。不是迷信，是"获得指引"的仪式感。

**交互设计**：
- 佛堂暗色背景 + 签筒摇动动画
- 用户点击开始摇签 → 竹签碰撞音效 → 点停止 → 出签
- 显示签图GIF（100张真实签图）
- 点"详细解签" → 完整解签内容（诗意/解曰/仙机/典故/各事项详解）

**数据来源**：
- 100签完整原文数据（sign-data.js，约300KB）
- 每签含：签级/宫位/典故/签文/诗意/解曰/仙机/精髓/完整详解/故事

**设计思考**：
- 为什么改名"问观音"不叫"求好运"？—— 不一定求好运，更像"问问指点迷津"
- 签文记录按用户隔离 —— 不同人的签不互相可见

### 模块5：我的

**定位**：个人中心 + 账号管理 + 数据安全。

**功能清单**：
- 圆形头像上传（居中裁剪200×200 + JPEG压缩）
- 自定义"快乐空间"名称
- 修改密码（云端+本地同步修改）
- 密保设置（5预设+自定义问题）
- 导出数据（JSON全量导出，含entries/shredded/bubblePool）
- 退出登录（清除CloudBase认证态+本地标记）

## 四、技术架构

### 整体架构
```
┌─────────────────────────────────────────┐
│           用户设备 (iOS/Android/PC)       │
├─────────────────────────────────────────┤
│  index.html (单文件，含全部CSS+JS+HTML)   │
│  sign-data.js (100签数据)                │
│  cloudbase.bundle.js (SDK 3.x)          │
│  sw.js (Service Worker离线缓存)          │
│  manifest.json (PWA配置)                 │
│  signs/ (100张签图GIF)                   │
├─────────────────────────────────────────┤
│  IndexedDB (本地持久化)                   │
│  ├── entries (快乐记录, _owner隔离)       │
│  ├── shredded (粉碎记录, _owner隔离)     │
│  └── bubblePool (泡泡池, _owner隔离)     │
├─────────────────────────────────────────┤
│  localStorage                            │
│  ├── hd_current_user (当前登录用户)       │
│  ├── hd_users (用户凭证表)               │
│  ├── hd_signs_{email} (签文历史)         │
│  ├── hd_avatar_{email} (头像Base64)     │
│  ├── hd_space_name_{email} (空间名)     │
│  └── hd_backup_remind_{email}           │
└─────────────────────────────────────────┘
                    ↕ 双向同步
┌─────────────────────────────────────────┐
│       腾讯云 CloudBase (19.9元/月)        │
│  ├── auth: 邮箱密码认证                   │
│  ├── database: entries (按_username隔离) │
│  ├── database: shredded                  │
│  └── database: bubblePool               │
└─────────────────────────────────────────┘
                    ↕ 静态文件
┌─────────────────────────────────────────┐
│       GitHub Pages (永久免费)             │
│  le4535441-star.github.io/happy-diary/   │
└─────────────────────────────────────────┘
```

### 关键设计决策
| 决策 | 选择 | 理由 |
|------|------|------|
| 前端架构 | 单文件HTML | 简单、零构建、直接部署 |
| 数据存储 | IndexedDB + CloudBase | 离线优先 + 多设备同步 |
| 前端托管 | GitHub Pages | 免费永久、无需备案 |
| 登录系统 | CloudBase邮箱密码 | 云端统一认证，多设备通用 |
| 图片存储 | Base64嵌入数据库 | 简单直接，无需额外对象存储 |
| 多账号隔离 | _owner字段过滤 | 最小改动实现数据安全 |

## 五、安全与数据保障

### 数据安全机制
1. **本地优先**：数据先存IndexedDB，断网也能用
2. **云端同步**：联网时自动双向同步到CloudBase
3. **多账号隔离**：_owner字段 + 读写过滤
4. **退出清认证**：doLogout调用auth.signOut()
5. **导出备份**：每月1-3号自动提醒导出JSON

### 数据持久性
- 前端代码：GitHub Pages永久免费托管
- 用户数据：CloudBase付费版（19.9元/月）
- 离线数据：IndexedDB（浏览器不清缓存就一直在）
- 备份机制：手动导出JSON文件

## 六、Token消耗分析与投入产出

### 本项目总体消耗估算

基于本次完整对话（从最初部署到所有bug修复完成），估算如下：

| 任务类别 | 预估轮次 | 预估Token消耗 | 占比 |
|---------|----------|--------------|------|
| 部署&基础设施（GitHub Pages/CloudBase/PWA） | ~15轮 | ~150K tokens | 20% |
| 核心功能开发（5大模块实现） | ~20轮 | ~200K tokens | 27% |
| Bug修复（签级/注册/闪屏/图片等） | ~15轮 | ~180K tokens | 24% |
| 多账号安全审查与修复 | ~8轮 | ~120K tokens | 16% |
| 需求讨论&方案确认 | ~10轮 | ~80K tokens | 11% |
| UI细节调整（字体/间距/tab大小等） | ~5轮 | ~20K tokens | 3% |
| **总计** | **~73轮** | **~750K tokens** | **100%** |

### Token价值估算
- 按当前模型定价（约 $15/MTok 输入 + $75/MTok 输出），本项目总消耗约 **$8-12**（≈¥60-85）
- 产出：一个完整的多功能PWA应用，含5大模块、多账号体系、云同步、离线支持

### 提升投入产出的建议

| 改进方向 | 预估节省 | 具体方法 |
|---------|---------|---------|
| **需求一次性给全** | 30%+ | 一开始给完整需求清单，而非逐条追加 |
| **避免反复调试** | 20% | 改完后先在本地预览确认，减少"推送→测试→发现问题→再修"的循环 |
| **减少重复上下文** | 15% | 长对话中跨session时，直接说"继续上次任务"并给关键词，不需要从头解释 |
| **一步到位的规格说明** | 10% | 对UI调整给具体数值（"字体改14px"而非"再大一点"），减少迭代 |

## 七、协作效率提升建议

### Doris 可以优化的点
1. **需求批量化**：把想到的功能先攒一批（3-5个），一次性给出优先级，我一次性做完
2. **测试后集中反馈**：测完一轮后把所有问题一次性列出来（比如"1.签级错 2.图片丢 3.字太小"），而非每发现一个就切回来
3. **对UI给具体数值**：比如"图标放大到28px"比"再大一点"更精确，省一轮确认
4. **信任一步到位**：如果我说"已修复已推送"，可以直接刷新看效果，不需要追问"都完成了吗"

### 我（AI）需要改进的点（已沉淀）
1. **首次实现就考虑多账号**：这次多账号隔离是事后补的，浪费了好几轮。以后做任何涉及数据存储的功能，第一次写就加上用户隔离
2. **主动预检而非被动修**：改完代码后主动跑一遍全局检查，不等用户测出问题
3. **Service Worker版本号自动化**：每次push前自动bump sw.js版本号，不要每次都手动记
4. **图片/大数据存储要提前考虑限制**：Base64存数据库有大小限制，第一次写就该加压缩+校验
5. **退出登录要完整**：不只是清localStorage，要清所有状态（认证态、内存变量、数据库连接）

### 最佳协作模式
```
Doris提需求（具体+有优先级）
    → 我直接动手（不问多余问题）
    → 实现+推送+告知
    → Doris真机测试
    → 集中反馈问题清单
    → 我一次性修完+推送
    → 确认OK，进入下一个需求
```
